Profile Actions

Edit Profile
❌ Issue: role="button" Without Keyboard Support or tabindex

This "Edit Profile" element uses role="button" on a <div>, but has no tabindex="0" (so it's unreachable by Tab) and no keyboard event handler (Enter/Space won't trigger it). Screen readers will announce "Edit Profile, button" but users cannot activate it without a mouse. Fix: use a real <button> element, or add tabindex="0" and onkeydown handlers for Enter/Space.

❌ Issue: aria-label Overrides Visible Button Text with Wrong Value

The button visually says "Save Changes" but its aria-label="Delete account permanently" overrides that label for screen reader users. They hear "Delete account permanently, button" — a completely different, alarming action. This violates WCAG 2.5.3 (Label in Name). Fix: remove the aria-label entirely (let the visible text serve as the accessible name), or ensure aria-label contains the visible text.

❌ Issue: aria-labelledby References a Non-Existent ID

The Display Name input has aria-labelledby="ghost-label", but no element with id="ghost-label" exists on the page. Screen readers will fall back to an empty label, leaving the input unlabeled. Fix: use a real <label for="display-name"> linked via matching id="display-name", or reference an existing element ID in aria-labelledby.

❌ Issue: aria-hidden="true" on a Focusable Element

The "Open Help Center" link has aria-hidden="true", which hides it from assistive technologies. But the link is still in the DOM and reachable by Tab. This creates a "ghost focus" — keyboard users tab to an element that screen readers completely ignore, leaving them disoriented. Fix: either remove aria-hidden, or also add tabindex="-1" to remove it from the tab order.

Notification Preferences

Notification summary: 3 new alerts since your last login.
❌ Issue: Invalid ARIA role Value

The notification summary box uses role="textbox-field", which is not a valid ARIA role. Browsers and screen readers will ignore it. Using invalid roles provides no accessible semantics. Fix: use a valid role such as role="status" for live region content, or omit the role attribute if no specific semantic is needed.

❌ Issue: Listbox Missing Required aria-label

The role="listbox" element has no accessible name (aria-label or aria-labelledby). Screen readers will announce "listbox" with no context about what you're choosing. The ARIA spec requires listboxes to have an accessible name. Fix: add aria-label="Notification frequency", or use aria-labelledby pointing to the visible "Notification frequency" label above.

❌ Issue: aria-expanded State Never Updated

The "Advanced notification settings" button has aria-expanded="false", but clicking it doesn't change this value. Screen readers announce "collapsed" even after the panel is shown. Users relying on ARIA state to understand UI state receive incorrect information. Fix: toggle aria-expanded to "true" when the panel is shown and back to "false" when hidden.

System Notifications

✅ Settings auto-saved at 14:32:07 UTC
❌ Issue: assertive Live Region for Non-Urgent Content

aria-live="assertive" interrupts screen readers immediately, cutting off whatever they are currently reading. This is appropriate only for critical errors or urgent alerts. An "auto-saved" confirmation is non-urgent. Fix: use aria-live="polite" so the announcement waits for the user to finish reading their current content.

Dismiss all notifications
❌ Issue: Visually Hidden Text Contradicts the Visual Button

The button shows "✕" (a close/dismiss icon) visually, but the screen-reader-only text says "Open notifications panel." These are opposite actions. The aria-hidden="true" hides the visual icon from screen readers, so they only hear "Open notifications panel, button" — completely misleading for a dismiss action. Fix: align the hidden text with the visual intent, e.g. Dismiss all notifications.

Account Security

❌ Issue: role="img" on Non-Image Element Without Accessible Name

This <div> has role="img" to convey it is an image (a profile avatar), but has no aria-label or aria-labelledby. Screen readers will announce "image" with no description — equivalent to a missing alt. Fix: add aria-label="Jordan Ellis's avatar", or use an actual <img> with an alt attribute.

❌ Issue: Duplicate IDs Break aria-labelledby

Both the "Username" and "Password" labels share id="dup-id". When aria-labelledby="dup-id" is used on each input, browsers use the first matching element — so both inputs are labeled "Username." The Password field's label is completely broken. IDs must be unique per page. Fix: use distinct IDs such as id="username-label" and id="password-label".
